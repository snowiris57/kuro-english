# プログラム作成シーケンス（git運用ルール）

> Claude Code用ワークフロー定義
> プログラムを作る・直すときは、このファイルの手順に従って動くこと。
> 中核思想：**「何を変えたか（what）」はdiffに残る。「なぜ変えたか（why）」はここにしか残らない。**

---

## 0. 大前提

- コミットメッセージの価値は「保存」ではなく「**判断の理由（why）を未来に残すこと**」。
- whyを残す相手は2人：**未来のゆうじさん**と、**次のセッションのAI（くろ / Claude Code）**。
- 履歴が判断の流れを語っていれば、ゆうじさんはAIの出力を追認するだけにならず、ハブを自分の手に残せる。
- これはObsidianの「一行メモ」のコード版。咲いた判断を、咲いたうちに一行残す。

---

## 1. 着手前

- [ ] そのリポジトリが `git init` 済みか確認。されていなければ初期化する。
- [ ] これから**何を作るか／なぜ作るか**を一行で宣言してから着手する。
- [ ] 既存コードを触る場合、まず現状の `git status` / `git log --oneline` を確認し、未コミットの変更がないかチェックする。

---

## 2. 変更は「目的の単位」で区切る

- **1つの目的 = 1コミット。** 「あれ直して」と「これ直して」を1コミットに混ぜない。
- ゆうじさんが複数の依頼をまとめて投げてきた場合でも、**内部では目的ごとに分けてコミットする**。
- 1コミットの粒度の目安：「このコミットを後で取り消したいと思ったとき、巻き戻したいのはこの変更だけか？」がYESになる単位。

---

## 3. コミットのたびに why を書く（最重要）

コミットメッセージは以下の形を基本とする：

```
<一行で「何をしたか」要約>

why: <なぜそう判断したか。代替案を退けた理由があれば添える>
```

- 1行目は簡潔に（whatの要約。diffを見れば分かる範囲でよい）。
- 本文の `why:` が主役。**「なぜこの方式にしたか」「なぜ他の選択肢を採らなかったか」を一行で残す。**
- 「リファクタした」「修正した」だけで終わらせない。理由のない変更はコミットしない。

### 良い例 / 悪い例

| | メッセージ |
|---|---|
| ❌ 悪い | `fix bug` |
| ❌ 悪い | `関数を修正` |
| ⭕ 良い | `キャッシュ参照を同期化 / why: 並列実行時に古い値を掴むレースが出たため。ロックより副作用が小さいと判断` |

---

## 4. 不具合修正のときは「現象→原因→対策」を残す

ゆうじさんの不具合解析スタイル（現象→原因仮説→物理メカニズム→検証）をコミットにそのまま写す。

```
<修正内容の要約>

現象: <何が起きていたか>
原因: <なぜ起きていたか（物理的/論理的な裏付け）>
対策: <なぜこの対策を選んだか>
```

- いきなり対策コミットに飛ばない。**原因の裏付けがメッセージにあること**を優先する。
- 「とりあえず動いた」で閉じない。動いた理由が分かっているなら、それも一行残す。

---

## 運用メモ

- このシーケンスは「プログラムを作る・直す」会話の冒頭で読み込ませる。常時SPに入れない（肥大化を避ける）。
- ゆうじさんが「今は試作だからコミットは粗くていい」と言ったら、3〜4の粒度をゆるめてよい。**体感フェーズと本番運用を分ける（動かす→硬くする）。**
- ただし「whyを一行残す」だけは、試作フェーズでも維持する。ここを省くと後で文脈が消える。
