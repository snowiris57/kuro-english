# 自己レビューの仕組み（/deep コマンド）

Sonnet（Fable より低コスト）で「作成 → 自己レビュー → 修正」を最大2周回してから結果を出す品質ループ。安いモデルに推論の手数を多く使わせることで、1回の応答より品質を引き上げる狙い。

## 使い方

Claude Code のセッションで：

```
/deep ボタンのタップ判定が小さいので44px以上に修正して
```

`/deep` の後ろに普通のタスクを書くだけ。コード変更・調査・文書作成いずれも可。

## 動作の流れ

```
ユーザー: /deep <タスク>
    │
    ▼
① 作業担当 (sonnet-worker) が成果物を作成
    │   「設計判断」「自信がない点」も申告させる
    ▼
② レビュー担当 (sonnet-reviewer) が批判的にレビュー
    │   別エージェントに分けているのは、自分の成果物に甘くなるのを防ぐため
    ▼
③ APPROVE → 完了報告
   REVISE  → 作業担当が指摘に1件ずつ対応（修正 or 根拠つき反論）
    │
    ▼
④ 再レビュー（2周目）→ 完了報告
   ※ 2周で打ち切り。残った指摘は報告に明記される
```

## ファイル構成

| ファイル | 役割 |
|---|---|
| `.claude/commands/deep.md` | `/deep` コマンド本体（ループの手順書） |
| `.claude/agents/sonnet-worker.md` | 作業担当エージェント（model: sonnet） |
| `.claude/agents/sonnet-reviewer.md` | レビュー担当エージェント（model: sonnet） |

## 期待値と限界

- 手順が明確なタスク（バグ修正、機能追加、文書化）では効果が出やすい。
- 一発で深い洞察が要る問題では Fable / Opus との差は残る。そういう時は素直に上位モデルを使う。
- サブエージェント実行もサブスクの使用量を消費する（1タスクあたり通常の2〜4倍程度のやり取りが発生する）。軽い作業には普通に依頼した方が安い。

## 調整したくなったら

- レビューの厳しさ・観点 → `.claude/agents/sonnet-reviewer.md` を編集
- ループ回数（現在2周上限） → `.claude/commands/deep.md` の手順5を編集
- モデルの変更（例：レビューだけ Opus にする） → 各エージェントの frontmatter `model:` を変更
