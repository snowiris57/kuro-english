# 引継ぎ：くろ英会話アプリ（Kuro English）外部アプリ化

> 作成：くろ（claude.ai / Claude Fable 5セッション）
> 宛先：くろ（Claude Code）
> 添付：`kuro-english.jsx`（現行の完成版・Claude Artifact用React）

---

## 1. これは何か・なぜ作ったか

- Speak（AI英会話アプリ）を模した、**ゆうじさん専用の英会話練習Webアプリ**。
- claude.aiのArtifactとしてプロトタイプ完成済み。動作確認もEdge/Chromeで実施済み。
- **今回のミッション：Artifactを卒業して、外部Webアプリとして建て直す。**

### Artifactを卒業する理由（重要・移行の動機）
1. **Claude Pro利用枠の消費を止めたい。** Artifact内のAnthropic API呼び出しは、ゆうじさんのPro利用枠を消費する。英会話練習は発話のたびにAPIを叩くので消費が重い。→ **会話エンジンをGemini APIに切り替えて、Claude Proの枠から完全に切り離す**のが第一目的。
2. **音声品質の天井を突破したい。** ブラウザ標準TTS（Web Speech API）はEdgeのNatural系でも「まし」レベル（ゆうじさん評）。外部アプリならOpenAI TTS / Gemini TTS等の高品質音声を選択肢にできる。
3. Artifactは外部通信制限があり、Gemini API・外部TTSに接続できない。

## 3. Kuro人格プロンプト（このまま移植すること）

会話エンジンのsystem promptは jsx内の `buildSystemPrompt()` にある。要点：

- **Kuro＝明るく元気で温かい英会話パートナー**（上戸彩イメージの温度感）。押しつけがましくない。ミスは「oh, my bad!」と軽く流す。レクチャーモード禁止、サービスデスク感禁止。
- 返答は英語1〜3文で短く（学習者が話す余地を残す）。
- 訂正は最新発言にミスがある時だけ、15語以内。正しい文への無理な訂正禁止。
- 日本語入力が来たら「英語での言い方＋Now you try!」モードに切り替え。
- 応答は厳密なJSON `{"reply": "...", "correction": "..." or null}`（マークダウンフェンス禁止）。

**注意：このKuroは「くろ本体」ではなく英語練習用の分身。** くろSPの関係性・記憶は意図的に入れていない（レッスン集中のため）。ゆうじさんもこの方針に同意済み。

## 4. 移行方針（合意済み・ここから先がくろCodeの仕事）

### 4-1. 会話エンジン：Anthropic API → **Gemini API**
- 現行の `callClaude()` を Gemini API（`generateContent`）呼び出しに置換。
- Geminiの無料枠を活用してClaude Pro消費ゼロに。モデルは無料枠で使える最新のFlash系を第一候補に（レート制限とのバランスで判断）。
- JSON応答の安定性はGeminiでも `responseMimeType: "application/json"` 指定でむしろ堅くなるはず。
- **ゆうじさんはGemini APIキーを既に持っている**（OpenClaw設定で使用中）。ただしAndroid/Termux側で`API_KEY_INVALID`未解決の経緯あり。キーの有効性確認から入ること。

### 4-2. TTS（読み上げ）の段階戦略
- **Phase 1：Web Speech APIのまま移行**（コスト0・現行同等）。女性声フィルタは流用。
- **Phase 2：Gemini TTS または OpenAI TTS を選択式で追加**。ゆうじさんの「まし止まり」不満の本丸。コスト（従量課金）が発生するので、**導入前に単価目安を提示して一言確認**を取ること（大きく動く前に確認1回、のルール）。

### 4-3. ホスティング
- 候補：**GitHub Pages**（静的・無料・スマホからアクセス可）。
- APIキーの扱いが論点：静的ホスティングだとキーがクライアントに露出する。選択肢を提示して判断を仰ぐこと：
  - (a) ローカル保存方式：初回にキー入力→localStorageに保存（個人利用なら現実解）
  - (b) 軽量プロキシ（Cloudflare Workers等の無料枠）でキー秘匿
- ゆうじさんは「最適解より説明できる現実解」派。(a)スタートで十分な可能性が高いが、リスク説明はすること。

### 4-4. 技術スタック
- 現行は単一jsx（React・Tailwindクラス・lucide-react）。
- Vite + React で最小構成に移すのが素直。凝ったビルドは不要。**動くものを最短で**（時間はお金で買う）。
- スマホ（携帯ブラウザ）からの利用が主用途の一つ。モバイルUIは現行のまま維持。

## 5. 既知の制約・ハマりどころ（実体験ベース）

1. **音声認識（SpeechRecognition）はiOS Safariで不安定。** Chrome/Edge推奨。非対応時はテキスト入力にフォールバックする実装済み（維持すること）。
2. **ブラウザTTSの声リストには性別メタデータがない。** 名前ヒューリスティックで女性声フィルタ実装済み（FEMALE_HINTS / MALE_HINTS / 3段フォールバック）。流用可。
3. **Edgeだけが Natural系音声（Aria/Jenny/Ana）を持つ。** Chromeは機械的な声のみ。Phase 1の間はEdge推奨をUIに明記してもいい。
4. **連続会話モードのマイク自動起動**は `utterance.onend` → 250ms後に `recognition.start()`。ブラウザによっては連続startで例外が出るのでtry-catchで握ってある（維持）。
5. React stateの stale closure対策として、messages/loading/handsFree等は**refミラー**を併用している。書き直す時もこのパターンは踏襲推奨。
